# Руководство по коннекторам AegisOps — Полная документация

## Что такое коннекторы?

**Коннекторы** — это программные модули-«мосты» между AegisOps и внешними информационными системами (1С, SAP, SCADA, Telegram, базы данных и т.д.). Каждый коннектор инкапсулирует всю логику взаимодействия с конкретным типом системы: знает, как подключиться, как передать данные, как получить ответ и как обработать ошибки.

**Простая аналогия:** Представьте AegisOps как умного диспетчера на центральном пульте газовой компании. Коннекторы — это телефонные линии и радиоканалы к каждому подразделению: компрессорной станции (OPC UA), бухгалтерии (1С), корпоративной ERP-системе (SAP), группе оповещения (Telegram) и так далее. Диспетчеру не нужно знать, как работает каждое подразделение изнутри — он просто использует нужную линию связи и формулирует запрос. Коннектор переводит запрос диспетчера в формат, понятный конкретной системе, и возвращает ответ в стандартизированном виде.

**Зачем нужны коннекторы, а не прямые запросы?** Без коннекторов каждый модуль AegisOps должен был бы самостоятельно реализовывать аутентификацию, обработку ошибок, таймауты, повторные попытки и форматирование запросов для каждой внешней системы. Коннекторы выносят эту логику в общий слой: модули просто вызывают `connector.fetchData()`, а коннектор сам заботится о деталях протокола, авторизации и устойчивости к сбоям.

### Как данные проходят через коннектор

```
1. Пользователь нажимает «Тест» / «Запустить сценарий» / «Запросить анализ»
       │
2. AegisOps определяет, какие коннекторы нужны для операции
       │
3. Connector Registry создаёт экземпляр нужного класса:
   new OllamaConnector(config) / new OpcUaConnector(config) / ...
       │
4. Вызывается метод коннектора:
   testConnection() — проверка доступности
   fetchData(query) — получение данных
   pushData(payload) — отправка данных
   discoverSchema() — обнаружение структуры
       │
5. Коннектор формирует HTTP/TCP запрос к внешней системе,
   используя safeFetch() с таймаутом и обработкой ошибок
       │
6. Результат возвращается в стандартизированном формате:
   { status: 'online'|'offline'|'error', data: ..., error: ... }
       │
7. AegisOps отображает результат пользователю или передаёт в AI
```

## Подробное описание каждого коннектора

### 1. Ollama (LLM — локальная нейросеть)

**Что делает:** Подключается к локальному серверу Ollama для генерации текста, анализа данных, ответов на вопросы и векторизации текста. Это «мозг» всей AI-платформы — без Ollama ни один AI-модуль не сможет работать.

**Какие задачи решает:**
- Генерация аналитических отчётов по газовому балансу
- Ответы на вопросы пользователей в AI-ассистенте
- Анализ рисков и прогнозирование потребления газа
- Векторизация текста для семантического поиска
- Fallback-анализ когда внешние системы недоступны

**Что нужно для подключения:**
- Ollama установлена на том же компьютере или доступна по сети
- Хотя бы одна модель скачана (рекомендуется `qwen2.5:7b-instruct` — 4.4 ГБ)
- Ollama работает на `http://127.0.0.1:11434` по умолчанию

**Как настроить в AegisOps:**
1. Откройте раздел «AI Движок» в боковом меню
2. Нажмите «Запустить всё автоматически» — система сама скачает и установит Ollama
3. Или добавьте коннектор вручную в разделе «Коннекторы»:
   - Имя: `Ollama Local`
   - Тип: `ollama`
   - Base URL: `http://127.0.0.1:11434`
   - Auth Mode: `none`

**Пример запроса через AI Ассистент:**
```
Пользователь: «Сформируй отчёт по газовому балансу за текущий месяц»
    │
AegisOps → OllamaConnector.chat({
  model: 'qwen2.5:7b-instruct',
  messages: [
    { role: 'system', content: 'Ты аналитик газовой компании...' },
    { role: 'user', content: 'Сформируй отчёт...' }
  ]
})
    │
Ollama → HTTP POST http://127.0.0.1:11434/api/chat
    │
Ответ нейросети отображается в чате
```

**API методы коннектора:**
- `chat(messages, model)` — генерация ответа в формате чата (основной метод)
- `embed(text, model)` — векторные представления текста для семантического поиска
- `listModels()` — список установленных моделей через `/api/tags`
- `showModel(name)` — информация о конкретной модели через `/api/show`

**Автоматические функции:**
- AegisOps автоматически определяет, установлена ли Ollama
- При нажатии «Запустить всё» — автоматически скачивает, устанавливает и запускает Ollama
- Список моделей обновляется автоматически
- Если Ollama недоступна, AI переключается на встроенный анализ по ключевым словам

### 2. 1C:Предприятие (OData)

**Что делает:** Подключается к 1С:Предприятие через стандартный OData-интерфейс. Позволяет читать справочники, документы, регистры сведений и накопления — все данные, которые администратор 1С разрешил для публикации.

**Какие задачи решает:**
- Выгрузка данных по потреблению газа из 1С:Бухгалтерия
- Чтение справочников контрагентов, номенклатуры, подразделений
- Получение документов реализации и поступления товаров
- Автоматическое обнаружение схемы данных через `$metadata`
- Фильтрация данных через OData-параметры: `$filter`, `$select`, `$expand`, `$top`, `$orderby`

**Что нужно для подключения:**
- 1С:Предприятие 8.3.6+ с опубликованным HTTP-сервисом OData
- URL публикации (например: `http://1c-server.company.local/erp/odata/standard.odata`)
- Логин и пароль пользователя 1С с правами на OData-доступ
- Сетевая доступность сервера 1С с компьютера, где установлен AegisOps

**Как настроить:**
1. **В 1С:** Администрирование → Публикация на веб-сервере → включите OData → выберите объекты для публикации (Справочники, Документы, Регистры)
2. **В AegisOps:** добавьте коннектор:
   - Имя: `1C Бухгалтерия`
   - Тип: `one_c_odata`
   - Base URL: `http://1c-server:8080/erp/odata/standard.odata`
   - Auth Mode: `basic`
   - Auth Payload: `{"login": "Администратор", "password": "ваш_пароль"}`

**Пример использования в сценарии:**
```
Сценарий: «Ежедневный отчёт по потреблению»
    │
1. ODataConnector.fetchData({
     entity: 'Document_РеализацияТоваровУслуг',
     $filter: "Date ge datetime'2025-01-01'",
     $select: 'Ref, Date, СуммаДокумента',
     $top: 1000
   })
    │
2. Данные передаются в Ollama для AI-анализа
    │
3. AI генерирует аналитический отчёт
```

**Особенности OData v3 (1С):**
- Имена сущностей с префиксом типа: `Catalog_Контрагенты`, `Document_РеализацияТоваров`
- Ключи в формате GUID: `guid'12345678-1234-1234-1234-1234567890ab'`
- Поддержка навигационных свойств через `$expand`
- Автоматический парсинг `$metadata` для обнаружения всех доступных сущностей

### 3. SAP S/4HANA (OData)

**Что делает:** Подключается к SAP через OData API. Читает бизнес-данные: заказы на продажу, инвойсы, финансовые проводки, данные о материалах и поставках.

**Какие задачи решает:**
- Синхронизация заказов на покупку и продажу с газовыми компаниями
- Получение финансовых данных для модуля «Финансово-экономические показатели»
- Анализ дебиторской и кредиторской задолженности через SAP-данные
- Мониторинг статусов заказов и поставок

**Что нужно для подключения:**
- SAP S/4HANA или SAP ECC с активированным OData сервисом
- URL сервиса (например: `https://sap-server:443/sap/opu/odata/sap/API_SALES_ORDER_SRV/`)
- SAP Username + Password или Bearer Token
- SAP Client (обычно 100, 400 — зависит от установки)
- Сетевой доступ к SAP-серверу (возможно через VPN)

**Как настроить:**
1. **В SAP:** активируйте OData сервис через транзакцию `/IWFND/MAINT_SERVICE`
2. **В AegisOps:** добавьте коннектор:
   - Имя: `SAP S/4HANA`
   - Тип: `sap_odata`
   - Base URL: `https://sap-server:443/sap/opu/odata/sap/API_SALES_ORDER_SRV/`
   - Auth Mode: `basic`
   - Auth Payload: `{"login": "SAP_USER", "password": "***"}`
   - Config: `{"sap-client": "100"}`

**Особенности SAP-коннектора:**
- Автоматическое получение CSRF-токена для операций записи (SAP требует токен для POST/PATCH/DELETE)
- Поддержка OData v2 (S/4HANA по умолчанию) и v4
- Автоматическая обработка пагинации через `__next`
- Заголовок `sap-client` для выбора манданта SAP

**Примеры стандартных API SAP для газового сектора:**
- `API_SALES_ORDER_SRV` — заказы на продажу
- `API_PURCHASE_ORDER_SRV` — заказы на покупку
- `API_OPLANNINGDATA_SRV` — данные планирования
- `API_COSTCENTER_SRV` — центры затрат

### 4. SCADA / OPC UA

**Что делает:** Подключается к промышленным SCADA-системам через протокол OPC UA. Читает показания датчиков в реальном времени: давление в трубопроводах, температуру газа, расход на узлах учёта, уровень в резервуарах. Может также записывать уставки для управления оборудованием.

**Какие задачи решает:**
- Мониторинг давления и температуры на газораспределительных станциях
- Чтение расхода газа с кориолисовых расходомеров
- Контроль уровня газа в резервуарах и газгольдерах
- Обнаружение аномалий в показаниях датчиков через AI-анализ
- Формирование прогнозов потребления на основе исторических данных SCADA

**Что нужно для подключения:**
- OPC UA сервер (встроен в большинство современных SCADA: WinCC OA, Ignition, Genesis64, zenon)
- Endpoint URL (например: `opc.tcp://scada-server:4840`)
- Если включена безопасность — сертификат клиента и/или логин/пароль
- Сетевая доступность OPC UA сервера (порт 4840 по умолчанию)

**Как настроить:**
1. Убедитесь, что OPC UA сервер запущен и доступен по сети
2. В AegisOps добавьте коннектор:
   - Имя: `SCADA Газокомпрессорная`
   - Тип: `opc_ua`
   - Base URL: `opc.tcp://192.168.1.100:4840`
   - Auth Mode: `none` (или `basic` если включена аутентификация)

**Что можно делать:**
- Читать значения узлов (Node IDs) — давление, температура, расход
- Обнаруживать структуру сервера (browse address space)
- Записывать значения (setpoint-управление) — только при наличии прав
- Автоматически обнаруживать схему через browse

**Пример использования:**
```
1. OpcUaConnector.discoverSchema()
   → Возвращает дерево узлов OPC UA сервера

2. OpcUaConnector.fetchData({
     nodeIds: [
       'ns=2;s=GasPipeline.Pressure',
       'ns=2;s=GasPipeline.Temperature',
       'ns=2;s=GasPipeline.FlowRate'
     ]
   })
   → Возвращает текущие значения датчиков

3. Данные передаются в AI для анализа аномалий
```

**Примечание по установке:** Коннектор использует библиотеку `node-opcua-client`, которая устанавливается автоматически при первом использовании. Если установка не удалась, проверьте наличие Visual Studio Build Tools (Windows) или build-essential (Linux).

### 5. Telegram Bot

**Что делает:** Отправляет сообщения, документы и уведомления в Telegram-чат или группу. Получает обновления (updates) от пользователей через бота. Это основной канал доставки отчётов и оповещений.

**Какие задачи решает:**
- Отправка ежедневных отчётов по газовому балансу
- Экстренные оповещения при аномалиях в давлении или потреблении
- Отправка PDF/Excel документов с аналитикой
- Интерактивное взаимодействие: пользователь пишет боту, AI генерирует ответ
- Отправка изображений с графиками и диаграммами

**Что нужно для подключения:**
- Создать бота через @BotFather в Telegram
- Получить Bot Token (например: `123456:ABC-DEF1234ghIkl-zyx57W2v1u123ew11`)
- ID чата или группы для отправки (можно получить через `getUpdates`)

**Как настроить:**
1. **В Telegram:** напишите @BotFather → `/newbot` → задайте имя и username → получите токен
2. **Добавьте бота** в группу или начните личный диалог с ним
3. **Узнайте chat_id:** отправьте боту любое сообщение, затем вызовите `getUpdates` — chat_id будет в ответе
4. **В AegisOps:** добавьте коннектор:
   - Имя: `Telegram Уведомления`
   - Тип: `telegram`
   - Base URL: `https://api.telegram.org`
   - Auth Mode: `token`
   - Auth Payload: `{"token": "123456:ABC-DEF..."}`

**API методы коннектора:**
- `sendMessage(chatId, text, parseMode)` — отправка текста с HTML/Markdown форматированием
- `sendDocument(chatId, fileUrl, caption)` — отправка файлов (PDF, Excel)
- `sendPhoto(chatId, photoUrl, caption)` — отправка изображений (графики, диаграммы)
- `getMe()` — проверка работоспособности бота
- `getUpdates()` — получение входящих сообщений

**Пример в сценарии:**
```
Сценарий: «Экстренное оповещение»
Условие: Давление < 2.0 бар
    │
TelegramConnector.sendMessage(
  chatId: '-1001234567890',
  text: '⚠️ ВНИМАНИЕ: Давление на ГРС-3 упало до 1.8 бар!',
  parseMode: 'HTML'
)
```

### 6. АСКУГ / UGaz / E-GAZ

**Что делает:** Подключается к системам автоматизированного контроля и учёта газа (АСКУГ), UGaz, E-GAZ для получения данных по потреблению, архивам, заправкам и биллингу. Это специализированный коннектор для газового сектора Узбекистана.

**Какие задачи решает:**
- Получение часовых архивов потребления газа по узлам учёта
- Чтение текущих показаний с узлов учёта в реальном времени
- Данные по заправкам на автомобильных газонаполнительных компрессорных станциях (АГНКС) через UGaz
- Биллинговая информация и начисления через E-GAZ
- Формирование прогнозов потребления на основе архивных данных

**Что нужно для подключения:**
- URL API системы (зависит от вендора):
  - Cometa: `http://cometa-server:8080/api/v1`
  - NEKTA: `http://nekta-server:8080/api/v1`
  - UGaz: `http://ugaz-server:8080/api/v1`
  - E-GAZ: `http://egaz-server:8080/api/v1`
- API-ключ или логин/пароль
- Идентификаторы узлов учёта (node IDs)

**Как настроить:**
1. Получите API-доступ у администратора системы учёта газа
2. В AegisOps добавьте коннектор:
   - Имя: `АСКУГ Ташкент`
   - Тип: `askug`
   - Base URL: `http://askug-server:8080/api/v1`
   - Auth Mode: `bearer` или `token`
   - Auth Payload: `{"token": "ваш_api_ключ"}`
   - Config: `{"vendor": "cometa", "api_version": "v1"}`

**Поддерживаемые вендоры и методы:**
| Вендор | Метод | Описание |
|--------|-------|----------|
| Cometa | `getHourlyArchive(nodeId, date)` | Часовой архив потребления |
| Cometa | `getCurrentReadings(nodeIds)` | Текущие показания узлов |
| NEKTA | `getHourlyArchive(nodeId, date)` | Часовой архив (NEKTA API) |
| UGaz | `getUgazRefuelData(stationId, from, to)` | Данные по заправкам на АГНКС |
| E-GAZ | `getEgazBilling(period)` | Биллинговые данные за период |

**Пример использования:**
```
1. AskugConnector.fetchData({
     endpoint: 'consumption',
     period: 'daily'
   })
   → Данные о потреблении за день

2. AskugConnector.getHourlyArchive('node-001', '2025-01-15')
   → Почасовые данные для узла node-001

3. AskugConnector.getUgazRefuelData('station-1', '2025-01-01', '2025-01-31')
   → Заправки на АГНКС за январь

4. Все данные → AI для прогнозирования
```

### 7. MQTT (Телеметрия / IoT)

**Что делает:** Получает данные телеметрии от IoT-устройств и датчиков через MQTT брокер. MQTT — лёгкий протокол обмена сообщениями, широко используемый в системах АСКУГ-ON-LINE, Tekinsoft и других IoT-платформах.

**Какие задачи решает:**
- Подписка на топики телеметрии в реальном времени
- Получение данных от газовых датчиков (расход, давление, температура)
- Интеграция с системами АСКУГ-ON-LINE, использующими MQTT
- Подключение к IoT-шлюзам Tekinsoft
- Агрегация данных телеметрии для AI-анализа

**Что нужно для подключения:**
- MQTT брокер (Mosquitto, HiveMQ, EMQX, Vernemq)
- Хост и порт брокера (стандартный порт 1883, WebSocket — 8080)
- Логин/пароль (если включена аутентификация)
- Топики для подписки (например: `gas/telemetry/#`, `gas/alerts/#`)

**Как настроить:**
1. Установите MQTT брокер (например: `sudo apt install mosquitto mosquitto-clients`)
2. В AegisOps добавьте коннектор:
   - Имя: `MQTT Телеметрия`
   - Тип: `mqtt`
   - Base URL: `mqtt://192.168.1.50:1883`
   - Auth Mode: `basic` (если нужна аутентификация)
   - Config: `{"topic": "gas/telemetry/#", "qos": 0}`

**Пример топиков для газового сектора:**
```
gas/telemetry/GRS-3/pressure     → 2.4 бар
gas/telemetry/GRS-3/temperature  → 18.5 °C
gas/telemetry/GRS-3/flow         → 1250 м³/ч
gas/alerts/GRS-3/low_pressure    → ALARM
gas/meters/node-001/reading      → 45678.9 м³
```

**Особенности работы:**
- Коннектор работает через WebSocket HTTP-bridge (порт 8080)
- Автоматически заменяет `mqtt://` на `http://` для REST-запросов к брокеру
- Поддерживает QoS 0 (at most once), 1 (at least once), 2 (exactly once)
- При отсутствии REST-bridge на брокере, можно использовать прямой MQTT-клиент (требует дополнительной настройки)

### 8. Базы данных (PostgreSQL / MySQL / MSSQL)

**Что делает:** Выполняет SQL-запросы к реляционным базам данных для извлечения аналитических данных. Поддерживает три наиболее распространённых СУБД: PostgreSQL, MySQL и Microsoft SQL Server (MSSQL).

**Какие задачи решает:**
- Прямые SQL-запросы к базам данных биллинговых систем
- Извлечение исторических данных потребления для AI-прогнозирования
- Агрегация данных из хранилищ (Data Warehouse)
- Обнаружение схемы базы данных для понимания структуры данных
- Подготовка данных для модулей газового баланса и финансов

**Что нужно:**
- Хост, порт, имя базы данных
- Логин/пароль с правами на чтение (рекомендуется только SELECT)
- Сетевая доступность СУБД

**Как настроить:**
1. Создайте пользователя только для чтения:
   ```sql
   -- PostgreSQL
   CREATE USER aegisops_readonly WITH PASSWORD 'secure_password';
   GRANT SELECT ON ALL TABLES IN SCHEMA public TO aegisops_readonly;

   -- MySQL
   CREATE USER 'aegisops_readonly'@'%' IDENTIFIED BY 'secure_password';
   GRANT SELECT ON gas_db.* TO 'aegisops_readonly'@'%';

   -- MSSQL
   CREATE LOGIN aegisops_readonly WITH PASSWORD = 'SecureP@ssw0rd';
   CREATE USER aegisops_readonly FOR LOGIN aegisops_readonly;
   ALTER ROLE db_datareader ADD MEMBER aegisops_readonly;
   ```
2. В AegisOps добавьте коннектор:
   - Имя: `PostgreSQL Газовые данные`
   - Тип: `database` (или `postgresql`, `mysql`, `mssql` для конкретной СУБД)
   - Base URL: `postgresql://192.168.1.10:5432/gas_db`
   - Auth Mode: `basic`
   - Config: `{"driver": "postgresql"}`

**Поддерживаемые драйверы:**
| Драйвер | Пакет | Описание |
|---------|-------|----------|
| `postgresql` | `pg` | PostgreSQL 9.5+ |
| `mysql` | `mysql2` | MySQL 5.7+ / MariaDB 10.2+ |
| `mssql` | `tedious` | Microsoft SQL Server 2012+ |

**Пример SQL-запроса:**
```sql
SELECT
  node_id,
  DATE_TRUNC('hour', reading_time) AS hour,
  AVG(volume) AS avg_consumption,
  MAX(pressure) AS max_pressure
FROM gas_readings
WHERE reading_time >= NOW() - INTERVAL '7 days'
GROUP BY node_id, DATE_TRUNC('hour', reading_time)
ORDER BY hour DESC;
```

### 9. REST API / GraphQL (Универсальный)

**Что делает:** Универсальный коннектор для подключения к любому REST или GraphQL API. Подходит для CRM-систем, ERP-платформ, внешних сервисов и любых HTTP-API, не имеющих специализированного коннектора.

**Какие задачи решает:**
- Интеграция с любыми REST API (CRM, ERP, GIS, GPS, мониторинг)
- GraphQL-запросы к современным API
- Пагинация результатов (offset, cursor, link header)
- JSON Path извлечение нужных полей из сложных ответов
- Подключение к Tekinsoft, Autocad, Seismic и другим системам через их REST API

**Что нужно:**
- URL API
- Метод аутентификации (API Key, Bearer Token, Basic Auth)
- Знание структуры API (endpoints, параметры) — обычно из документации API

**Как настроить:**
1. Получите API-документацию у провайдера системы
2. В AegisOps добавьте коннектор:
   - Имя: `CRM REST API`
   - Тип: `crm_rest` (или `erp_rest`, `rest`, `graphql`)
   - Base URL: `https://crm.example.com/api/v2`
   - Auth Mode: `bearer`
   - Auth Payload: `{"token": "your-api-key"}`
   - Config: `{"pagination": "offset", "jsonPath": "$.data.items"}`

**Особенности:**
- Автоматическая обработка пагинации: если в ответе есть `next`/`__next`/Link header, коннектор сам перейдёт на следующую страницу
- Поддержка GraphQL: отправка запросов в формате `{ query: "...", variables: {...} }`
- JSON Path извлечение: если API возвращает вложенные данные, можно указать путь к нужному массиву

**Примеры типов коннекторов на базе REST:**
| Тип | Описание | Типичное использование |
|-----|----------|----------------------|
| `crm_rest` | CRM через REST | Клиенты, сделки, контакты |
| `erp_rest` | ERP через REST | Заказы, поставки, склад |
| `rest` | Универсальный REST | Любой HTTP API |
| `graphql` | GraphQL endpoint | Современные API |
| `tekinsoft` | Tekinsoft API | Газовые системы Tekinsoft |
| `gis` | GIS REST API | Геоинформационные системы |
| `gps` | GPS tracking API | Мониторинг транспорта |

## Пошаговая инструкция «Как подключить коннектор»

### Шаг 1: Определите тип системы
Ответьте на вопрос: к какой системе вы хотите подключиться?
- Бухгалтерия/ERP на 1С? → Тип `one_c_odata`
- Корпоративная ERP на SAP? → Тип `sap_odata`
- Промышленная SCADA? → Тип `opc_ua`
- Уведомления в Telegram? → Тип `telegram`
- Система учёта газа (АСКУГ)? → Тип `askug`
- IoT телеметрия? → Тип `mqtt`
- База данных? → Тип `database`
- Любой другой API? → Тип `rest`
- Email рассылка? → Тип `email`
- Webhook/Slack/Teams? → Тип `webhook`

### Шаг 2: Соберите информацию
Вам нужно знать:
- **URL доступа** к системе (http://..., opc.tcp://..., mqtt://..., smtp://...)
- **Учётные данные** (логин/пароль, API-токен, сертификат)
- **Сетевую доступность** — можно ли достичь сервера с компьютера AegisOps (проверьте через `ping` или `curl`)
- **Документацию API** — какие методы доступны, какие параметры принимают

### Шаг 3: Добавьте коннектор в AegisOps
1. Откройте раздел **«Коннекторы»** в боковом меню
2. Нажмите **«+ Добавить коннектор»**
3. Заполните форму:
   - **Имя** — любое понятное название (например: «1С Бухгалтерия Ташкентгаз»)
   - **Тип** — выберите из выпадающего списка
   - **Base URL** — адрес системы
   - **Auth Mode** — метод аутентификации (none/basic/bearer/token/api_key)
   - **Auth Payload** — учётные данные в формате JSON
4. Нажмите **«Сохранить»**

### Шаг 4: Протестируйте подключение
1. Нажмите кнопку **«Тест»** рядом с коннектором
2. AegisOps попробует связаться с системой
3. Если статус **«online»** — всё работает! Можно переходить к шагу 5
4. Если **«offline»** — проверьте:
   - Правильность URL
   - Учётные данные
   - Сетевую доступность (попробуйте `curl http://ваш-url` в терминале)
   - Запущена ли внешняя система

### Шаг 5: Используйте в сценариях
1. Создайте сценарий в разделе **«Сценарии»**
2. Укажите коннекторы, которые нужно опросить
3. Опишите цель сценария (AI поймёт, что нужно сделать с данными)
4. Запустите сценарий — AegisOps:
   - Опросит все указанные коннекторы
   - Соберёт реальные данные
   - Отправит данные в AI (Ollama) для анализа
   - Сгенерирует отчёт и/или отправит уведомление

## Технические детали работы коннекторов

### BaseConnector — базовый класс

Все 10 коннекторов наследуются от абстрактного класса `BaseConnector`, который предоставляет общий каркас:

```javascript
class BaseConnector {
  constructor(config) {
    this.id = config.id;
    this.name = config.name;
    this.type = config.type;
    this.baseUrl = config.base_url || config.baseUrl;
    this.authMode = config.auth_mode || 'none';
    this.authPayload = config.auth_payload || {};
    this.config = config.config || {};
    this.enabled = config.enabled !== false;
    this.timeout = config.timeout || 15000; // 15 секунд по умолчанию
  }

  // Обязательные методы (переопределяются в каждом коннекторе)
  async testConnection()  { throw new Error('Not implemented'); }
  async fetchData(query)  { throw new Error('Not implemented'); }
  async pushData(payload) { throw new Error('Not implemented'); }

  // Опциональные методы
  async discoverSchema()  { return { entities: [] }; }

  // Вспомогательные методы
  getAuthHeaders() { /* строит заголовки для basic/bearer/token/api_key */ }
  async safeFetch(url, options) { /* HTTP с AbortController таймаутом */ }
}
```

### Жизненный цикл коннектора

```
┌─────────────────────────────────────────────────┐
│ 1. СОЗДАНИЕ                                     │
│    Пользователь добавляет коннектор через UI     │
│    → Конфигурация сохраняется в SQLite           │
│    → Коннектор НЕ создан в памяти                │
├─────────────────────────────────────────────────┤
│ 2. ИНСТАНЦИАЦИЯ                                 │
│    При запросе (тест/данные/сценарий)            │
│    → Connector Registry читает конфиг из БД      │
│    → Создаёт экземпляр: new OllamaConnector(cfg) │
│    → Вызывает нужный метод                       │
├─────────────────────────────────────────────────┤
│ 3. ВЫПОЛНЕНИЕ ЗАПРОСА                           │
│    → getAuthHeaders() формирует заголовки        │
│    → safeFetch() выполняет HTTP-запрос           │
│    → AbortController обеспечивает таймаут        │
│    → Если ошибка — возвращается структура с      │
│      status:'offline' и suggestion               │
├─────────────────────────────────────────────────┤
│ 4. ВОЗВРАТ РЕЗУЛЬТАТА                            │
│    → Стандартный формат:                         │
│      { status, data?, error?, suggestion? }      │
│    → UI отображает результат                     │
│    → Или AI использует данные для анализа        │
├─────────────────────────────────────────────────┤
│ 5. УДАЛЕНИЕ                                      │
│    Пользователь удаляет коннектор через UI       │
│    → Конфигурация удаляется из SQLite            │
│    → Экземпляр больше не создаётся               │
└─────────────────────────────────────────────────┘
```

### Connector Registry (Фабрика коннекторов)

Connector Registry (`connectors/index.js`) — это фабрика, которая по строковому типу создаёт нужный экземпляр:

```javascript
// Когда AegisOps нужен коннектор:
const connector = createConnector({ type: 'opc_ua', base_url: '...', ... });

// Registry определяет класс:
// 'opc_ua' → OpcUaConnector
// 'one_c_odata' → ODataConnector
// 'askug' → AskugConnector
// и т.д.

// Затем вызывается метод:
const result = await connector.testConnection();
```

### Обработка ошибок

Каждый коннектор **не выбрасывает исключения** при ошибке подключения. Вместо этого он возвращает структурированный ответ с рекомендацией:

```json
// Успешное подключение
{
  "status": "online",
  "data": { ... },
  "vendor": "cometa",
  "endpoint": "http://..."
}

// Ошибка подключения
{
  "status": "offline",
  "error": "Connection refused",
  "suggestion": "Убедитесь что Ollama запущена: ollama serve"
}

// Ошибка аутентификации
{
  "status": "error",
  "error": "HTTP 401 Unauthorized",
  "suggestion": "Проверьте логин и пароль в настройках коннектора"
}
```

Это позволяет AI-движку продолжать работу даже когда часть систем недоступна. Например, если SCADA-сервер временно перезагружается, AI использует fallback-анализ на основе данных из других источников (1С, базы данных).

### Безопасность

- **Учётные данные** хранятся в локальной SQLite базе данных (на том же компьютере)
- **API-ключи** хешируются с помощью scrypt (одностороннее хеширование — пароль нельзя восстановить)
- **JWT-токены** для админ-доступа имеют срок действия 24 часа
- **Localhost bypass** — запросы с 127.0.0.1 не требуют аутентификации (отключаемо в настройках)
- **Rate Limiting** — максимум 300 запросов в минуту на API для защиты от перегрузки
- **HMAC-SHA256 подпись** — для вебхуков, если указан секрет
- **Таймауты** — все запросы к внешним системам имеют таймаут 15 секунд (настраиваемо)

## Частые вопросы (FAQ)

**Q: Нужно ли устанавливать драйверы для коннекторов?**
A: Большинство коннекторов работают через HTTP и не требуют дополнительного ПО. Исключения:
- OPC UA: npm автоматически устанавливает `node-opcua-client` при первом использовании (требует C++ компилятор)
- Базы данных: драйверы `pg`, `mysql2`, `tedious` устанавливаются вместе с AegisOps
- Email: `nodemailer` входит в стандартную установку

**Q: Что если внешняя система в другой сети (другой офис, VPN)?**
A: Настройте VPN между сетями или используйте встроенный Cloudflare Tunnel (раздел «Мобильный доступ»). Туннель создаёт безопасное соединение через интернет без открытия портов.

**Q: Можно ли подключить несколько систем одного типа (например, две 1С)?**
A: Да! Добавляйте несколько коннекторов одного типа, каждый со своим URL и учётными данными. Например: «1С Бухгалтерия» и «1С ERP» — это два отдельных коннектора типа `one_c_odata`.

**Q: Что делать если тест коннектора показывает «offline»?**
A: Пошаговая диагностика:
1. Проверьте, что сервер внешней системы запущен
2. Проверьте URL — нет ли опечатки
3. Проверьте сеть: `ping server-name` и `curl http://server:port/health`
4. Проверьте учётные данные — логин, пароль, API-ключ
5. Проверьте брандмауэр — не блокирует ли порт
6. Если используется HTTPS — проверьте сертификат

**Q: Могу ли я написать свой коннектор?**
A: Да! Создайте файл в `server/connectors/`, унаследуйте от `BaseConnector` и зарегистрируйте в `connectors/index.js`. Минимальный пример:
```javascript
const { BaseConnector } = require('./base');
class MyConnector extends BaseConnector {
  async testConnection() {
    try {
      const res = await this.safeFetch(this.baseUrl + '/health', {});
      return { status: res.ok ? 'online' : 'error' };
    } catch (err) {
      return { status: 'offline', error: err.message };
    }
  }
  async fetchData(query) { /* ... */ }
  async pushData(payload) { /* ... */ }
}
module.exports = { MyConnector };
```

**Q: Как коннекторы работают с AI-модулями?**
A: Когда вы запускаете модуль (например, «Газовый баланс»), AegisOps:
1. Определяет, какие коннекторы могут предоставить нужные данные
2. Опрашивает их через `fetchData()`
3. Объединяет данные из всех источников
4. Формирует контекст для AI-модели (Ollama)
5. AI анализирует данные и генерирует отчёт

**Q: Безопасно ли хранить пароли в AegisOps?**
A: Да, при соблюдении базовых правил:
- AegisOps работает локально на вашем компьютере — данные не уходят в интернет
- API-ключи хешируются (нельзя прочитать обратно)
- Для подключения к AegisOps нужна аутентификация (JWT или API-ключ)
- Рекомендуется использовать read-only учётные записи для баз данных
- Для 1С и SAP создавайте отдельных пользователей с минимальными правами

**Q: Можно ли использовать AegisOps без Ollama (без AI)?**
A: Да, но с ограничениями. Без Ollama:
- Коннекторы будут работать — можно тестировать, получать данные
- Сценарии будут выполняться, но вместо AI-анализа будет использоваться встроенный анализ по ключевым словам
- AI Ассистент будет недоступен
- Рекомендуется установить Ollama — это бесплатно и работает на обычном компьютере

**Q: Сколько коннекторов можно добавить?**
A: Технических ограничений нет. Практически — ограничено только ресурсами компьютера (память, сетевые подключения). Рекомендуется не более 20-30 активных коннекторов на одной установке.

**Q: Работают ли коннекторы через прокси?**
A: Да. Установите переменные окружения `HTTP_PROXY` и `HTTPS_PROXY` перед запуском AegisOps. Коннекторы используют `fetch()`, который автоматически подхватывает настройки прокси.
